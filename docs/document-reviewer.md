# 문서 검토 드로어 — 인사팀용 "누가·어디서" 패널

> 목표: **인사팀(비개발자)이 자동 추출 결과를 원문과 나란히 빠르게 검증**한다. 관계자 한 명이
> 어느 문서 몇 쪽에서, 어떤 문맥으로 나왔는지를 클릭 한 번으로 원문에서 확인할 수 있어야 한다.
> (SRE 관점: 행정 검증 시간을 줄이는 게 핵심 가치. 화려함보다 "클릭 → 바로 확인"의 신뢰성.)

## 벤치마킹 — 상용 서비스에서 가져온 패턴

| 서비스 | 가져온 패턴 |
|---|---|
| **Greenhouse / Lever (ATS)** | 채용담당자가 이력서에서 매칭 키워드를 하이라이트로 본다 — 우리의 "검출 근거" 하이라이트와 동일 목적 |
| **Mozilla pdf.js 뷰어** | `#page=N` 딥링크 페이지 이동 + find controller 텍스트 하이라이트 |
| **Adobe PDF Embed API** | 페이지 이동 + 검색어 강조 + 주석 오버레이 |
| **Notion 사이드 피크** | 우측 슬라이드 패널로 맥락 전환 없이 원문 열람(이미 채택) |
| **GitHub PR 리뷰** | 좌측 목록(파일/사람) ↔ 우측 본문, 항목 클릭 시 해당 위치로 점프 |

핵심 결론: **좌측 "사람 목록" + 우측 "원문", 항목 클릭 → 원문 해당 쪽 점프**가 비개발자에게 가장
직관적이다. 하이라이트는 그 위에 얹는 강화 단계.

## Phase 1 — 점프 + 근거 패널 (이번 구현, 무빌드/저위험)

- 문서 카드 클릭 → 우측 드로어가 **2단**으로 열린다.
  - **좌측**: "이 문서에서 검출된 관계자 N명" — 이름(+한글 추정), 역할 배지, **`p.N` 페이지 칩**,
    검출 근거 스니펫(`evidence`), 본인/미확인 배지.
  - **우측**: 원문(PDF 내장 뷰어 / 이미지 / 추출 텍스트).
- **이름 클릭** → 우측 PDF가 그 사람의 **첫 출현 페이지**로 이동. **`p.N` 칩 클릭** → 해당 쪽으로 이동.
  - 구현: PDF는 `/api/file/{id}#page=N` 앵커로 이동(브라우저 내장 뷰어가 지원, 파일 라우트가 inline 서빙).
    `key`로 iframe을 리마운트해 같은 문서 내 재점프도 확실히 동작시킨다.
- 데이터는 이미 있는 `person_aggregates.sources[]`(documentId·page·evidence)에서 읽는다 — **DB 변경 없음**.

## Phase 2 — 원문 하이라이트(형광펜) (차기, pdf.js 클라이언트 렌더)

페이지 점프까지는 내장 뷰어로 충분하지만, **이름 위치 형광펜**은 브라우저 내장 뷰어가 지원하지 않아
직접 렌더가 필요하다.

- 대상 페이지를 `pdfjs-dist`로 캔버스 + **text layer** 렌더 → `evidence`/이름 문자열을 text layer에서
  찾아 위치 사각형에 하이라이트 오버레이.
- 비용: 클라이언트 pdf.js 번들(worker·폰트·CSP), 큰 문서(예: 181쪽 학위논문)는 **해당 페이지만** 렌더해
  메모리 억제. 점프는 Phase 1 앵커 유지, 하이라이트는 "검출 위치" 미니 뷰로 병행 제공.
- 도장·서명은 이미 bbox가 있어(detect-marks/crop) 크롭 갤러리로 보여주는데, 텍스트 인물은 bbox가 없으므로
  text-layer 문자열 매칭으로 위치를 역산한다.

## Phase 3 — 추출 진행 표시 (이번 구현에 포함)

- 추출 중에는 텍스트 "running" 대신 **회전 스피너 + "추출 중…" 배지**.
- 진행 중이면 클라이언트가 일정 간격으로 `router.refresh()` 해 **자동 갱신**(완료 시 중단, ~10분 안전상한).
  인사팀이 수동 새로고침 없이 결과가 채워지는 걸 본다.
